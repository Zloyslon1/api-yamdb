# YaMDb — план командной работы

> Это наш рабочий документ. Главный источник правды по API — файл **`api_yamdb/static/redoc.yaml`** (локально открывается на http://127.0.0.1:8000/redoc/).
> Если код вдруг расходится с редоком — значит, прав редок, а не код. Когда сомневаетесь — сначала смотрим документацию, потом тесты, и только потом спорим.

## 0. Когда задача считается готовой

Я предлагаю договориться так: задача закрыта, только когда выполнены **все три** пункта сразу:

1. **Тесты зелёные** — гоняем `pytest` по своей зоне (какие именно тесты ваши — см. §7).
2. **`flake8` молчит** — настройки лежат в `setup.cfg`. Папку `tests/` при этом не трогаем.
3. **Совпадает с редоком** — имена полей, методы, права и коды ответов как в документации.

Если хоть один пункт не выполнен — задача ещё в работе, даже если «вроде всё работает».

## 1. Git-workflow (это важно, прочитайте все)

**Самое главное правило: напрямую в `main` мы не пушим. Вообще никогда.** В `main` код попадает только через Pull Request, и только после апрува.

### Ветки
- `main` — защищённая, тут всегда рабочий код.
- `feature/<что-делаем>` — ветка под конкретную задачу.
- Одна задача = одна ветка = один PR.

### Как работать с ветками
```bash
git checkout main
git pull                          # подтянули свежий main
git checkout -b feature/моя-задача
# ...пишем код, коммитим почаще и по делу...
git push -u origin feature/моя-задача
# и открываем Pull Request в main на GitHub
```

### Про PR
- Старайтесь делать PR **поменьше**
- В описании коротко напишите: что сделал, что стоит проверить, какие тесты проходят.
- **Свой PR сам не мёржишь** — нужен апрув хотя бы одного человека из команды.
- Перед тем как открыть PR, влейте себе свежий main: `git merge main`.

### Защита main (это я уже настроил)
- ✅ Require a pull request before merging
- ✅ Require approvals: 1
- ✅ Require conversation resolution before merging
- ✅ Do not allow bypassing — правила одинаковые для всех, тимлид тоже не исключение.

## 2. Как устроен проект

У нас два Django-приложения:

| Приложение | За что отвечает |
|---|---|
| `reviews` | Все **модели** + админка + management-команда для импорта CSV |

## 3. Модель данных

```
Category (name, slug)                 Genre (name, slug)
      \                                   /
       \  FK (SET_NULL)        M2M (genre_title)
        \                         /
         ──────  Title (name, year, category, genre[], *rating)
                          │
                          │ FK (CASCADE)
                          ▼
User ───FK(CASCADE)──► Review (title, author, text, score 1-10, pub_date)
  │                        │
  │                        │ FK (CASCADE)
  └───FK(CASCADE)────► Comment (review, author, text, pub_date)
```

`rating`: это поле **в базе не хранится**. Считаем его как среднее `score` по всем отзывам (`annotate(Avg('score'))`).

### Что происходит при удалении (`on_delete`)
| Удаляем | Что происходит | Чем сделано |
|---|---|---|
| User | вместе с ним уходят его reviews и comments | FK `CASCADE` |
| Title | удаляются его reviews и все их comments | FK `CASCADE` |
| Review | удаляются его comments | FK `CASCADE` |
| Category | произведение остаётся, поле `category` становится пустым | FK `SET_NULL, null=True` |
| Genre | произведение остаётся, просто пропадает связь | M2M (через `genre_title`) |

### Ограничения
- **Один автор — один отзыв на произведение**: `UniqueConstraint(fields=['title', 'author'])`, повторный отзыв вернёт `400`.
- `score` — целое число строго **1..10** (вешаем валидатор).
- `year` — **не позже текущего** (тоже валидатор).

## 4. Пользователи и роли

**При регистрации (signup) просим только** `email` и `username`.
**В профиле полей больше:** `username`, `email`, `first_name`, `last_name`, `bio`, `role`.

| Роль | Что может |
|---|---|
| Аноним | только читать titles/reviews/comments |
| `user` (роль по умолчанию) | + создавать/редактировать/удалять **свои** отзывы и комменты |
| `moderator` | + редактировать/удалять **любые** отзывы и комменты |
| `admin` | всё: titles/categories/genres + раздавать роли |

**Важный момент про суперюзера.** Суперюзер всегда админ, и права ему даёт флаг `is_superuser`, а не поле `role`. Поэтому проверку прав делаем как `is_superuser or role == 'admin'`. Даже если суперюзеру руками выставить `role='user'`, он всё равно останется суперюзером — на это легко наступить, имейте в виду.

**Валидации:** `username` по паттерну `^[\w.@+-]+\Z`, не длиннее 150 символов и **не может быть `me`**. `email` — до 254 символов.

## 5. Аутентификация (тут вообще черт ногу сломит)

```
1. POST /api/v1/auth/signup/      {email, username}     → 200, на почту уходит confirmation_code
2. POST /api/v1/auth/token/       {username, confirmation_code} → 200 {token}  (JWT)
3. Дальше все запросы с заголовком: Authorization: Bearer <token>
4. PATCH /api/v1/users/me/        чтобы обновить свой профиль
```

Важное из редока:
- `signup` возвращает **200**
- **`signup` можно использовать повторно** — если такой юзер уже есть — просто высылаем код заново.
- В `token`: если юзера не нашли → **404**; если код неверный или поля нет → **400**.
- Письма шлём через console email backend, так что код можно подсмотреть прямо в консоли (или в файле).

### Когда юзера создаёт админ
`POST /api/v1/users/` (доступно только админу) создаёт юзера и **может сразу выставить ему роль**. Письмо при этом **не отправляется**: юзер потом сам идёт на `signup` за кодом.

## 6. Импорт CSV

Делаем management-командой (`python manage.py import_csv`), которая грузит файлы из `api_yamdb/static/data/*.csv`.

**Порядок загрузки важен — из-за внешних ключей:**
```
category, genre  →  titles  →  genre_title (M2M)  →  users  →  review  →  comments
```
Данные там специально косячные, как я понял — попадаются пропуски и дубли, обработку ошибок закладываем сразу. Файл `genre_title.csv` — это промежуточная таблица для связи M2M.

## 7. Кто что делает

### Фаза 0 — фундамент (делаем до того как разбежимся по задачам, и мёржим в main)

- структура приложений `reviews` + `api`;
- `settings.py`: DRF, SimpleJWT, `AUTH_USER_MODEL`, email backend в консоль, пагинация;
- **кастомная модель `User` с полем `role`** — это критично сделать **до самой первой миграции**, потому что `AUTH_USER_MODEL` потом уже нормально не поменять;
- заглушки пермишенов (`IsAdmin`, `IsAdminOrReadOnly`, `IsAuthorOrModeratorOrAdmin`);
- корневой `api/urls.py` с роутером — каждый потом просто регистрирует туда свой ViewSet.

### Фаза 1 — расходимся и пилим параллельно

| Зона | Что трогаем | Задачи | «Готово» |
|---|---|---|---|
| **A** | auth + users | signup/token/confirmation_code, отправка письма; users CRUD (для админа), `me/`, назначение ролей, поиск по username, пагинация | `test_00`, `test_01` |
| **B** | titles + categories + genres | модели Category/Genre/Title; slug-эндпоинты categories/genres; **фильтрация titles** (category, genre, name, year); `rating` через annotate; два разных сериализатора Title (на чтение — вложенные объекты, на запись — slug) | `test_02`, `test_03`, `test_04` |
| **C** | reviews + comments | вложенные роуты `/titles/{id}/reviews/...`; модели со `score`; `UniqueConstraint(title, author)`; пермишены автор/модератор/админ; автоподстановка автора и произведения | `test_05`, `test_06` |
| **общее** | импорт CSV | management-команда (см. §6) — берёт Dev B или кто первый освободится | `test_07` |

> **Важно:** `Review` (зона C) опирается на `Title` (зона B). Поэтому, **Dev B первым делом делает модель `Title`** и быстро её мёржит, чтобы не держать C. А пока `Title` нет — C спокойно пишет то, что от него не зависит: пермишены и структуру сериализаторов.
